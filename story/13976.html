作家如何使用GitHub存储他们的作品

有很多方法可以管理和存储您的写作项目。一些人更喜欢云存储服务(如Dropbox)或在线编辑器(如Google Docs)，而另一些人则使用桌面应用程序(如Microsoft Word)。我用的是一种叫GitHub的东西。

GitHub：它不仅仅用于编码

我使用Git和GitHub来存储和访问我所有的作品。Git是一个有效的工具，你可以用来跟踪文档的变化，而且你还可以超快地上传到GitHub。将您的作品下载到第二个或第三个设备也是快速而简单的。

如果您从未听说过GitHub，那么它是世界上最流行的存储和维护开源代码的目的地。那可能听起来像是一个疯狂的写作场所，但事实并非如此！毕竟，代码只是一行又一行的文本，就像你的文章、故事或论文一样。

大约在2013年，GitHub开始鼓励人们为所有类型的信息创建存储库，而不仅仅是代码。GitHub从未真正离开过它的编码根源，但一些人仍然使用它来存储写作和其他非编码项目。例如，一个人用Git和GitHub写了一本教学书籍，而另一个人写了一本小说。在谷歌上转转，你会发现GitHub的各种疯狂用途。

什么是Git和GitHub？

Git是由Linux名人Linus Torvalds创建的开源程序。Git跟踪文档的更改，使多人可以更轻松地远程处理同一文档。用技术术语来说，它被称为分布式版本控制系统(或分布式VCS)。Git不会以设定的时间间隔任意保存文档的版本。相反，只有当您告诉它时，它才会存储对文档的更改。

您的文档形成存储库(或repo)，这只是项目文件夹的一个花哨术语。例如，如果您使用Git来管理您在Windows中的Documents文件夹，那么它将是一个存储库(但不要这么做)。

当您在Git中存储对文档的更改时，这称为“提交”。提交只是您对文档所做的最新更改的记录。每次提交都会分配一个由数字和字母组成的长字符串作为其ID。

如果按ID调用过去的提交，则不会像在Word的文档历史记录中那样看到整个项目。您只能在提交时看到最新的更改。然而，这并不意味着整个项目没有被记录下来。您可以删除项目文件夹中的所有内容，但仍然可以使用几个git命令取回最新版本。你甚至可以回去看看这个项目在一周前或六个月前是什么样子。

您还可以在每次提交时包含消息，这非常有用。例如，如果您编写了一些内容，但不确定是否要保留它，只需提交即可。即使您稍后将其从项目中删除，该节也会保留在您的提交历史记录中。

Git在命令行上工作得最好，这是一个很大的优势，但也有其缺点。命令行可以很好地创建、提交和上载更改。但是，如果您想要查看提交历史记录，这并不理想。

这就是为什么很多人喜欢GitHub的原因--GitHub是一个流行的在线服务，它为你的Git存储库提供了一个网络界面。在GitHub上，你可以很容易地查看过去的提交，以及将你的作品下载到多台PC上。

Git和GitHub一起让我可以在粒度级别上控制我的版本历史。而且，在任何可以运行Bash命令行的PC上都很容易写出我的东西，这些天来，包括Windows、Mac、Linux和Chrome OS机器。

纯文本文件让事情变得简单

Git和GitHub几乎支持任何文件类型的编写，尽管它在纯文本上工作得最好。如果您用Microsoft Word编写，它会起作用，但您将无法在命令行或GitHub中查看您过去的提交。相反，您必须在命令行调用过去的提交(称为“签出”)，然后打开Word文件。然后，Word文件看起来就像您进行原始提交时一样，您可以使用另一个快速命令返回到当前版本。

如果您使用Scrivener，也可以。Scrivener将文件保存为文本，因此它还会在GitHub和命令行上显示过去的提交。但是Scrivener也会保存对程序重要但对您不重要的数据。在每一次提交中，您都会得到大量的垃圾信息，这使得阅读变得很困难。

我使用纯文本文件，因为这是您将单词串在一起所需的全部内容，特别是在最初的几个草稿中。

让我们来了解一下这一切是如何工作的技术细节。我们将从PC开始，然后使用GitHub升级到云。

要开始使用，您需要MacOS或Linux上的终端程序。如果您的计算机运行Windows10，则必须通过Windows Subsystem for Linux(WSL)安装Ubuntu或其他Linux发行版，这非常容易。您可以查看我们关于如何在Windows10上安装Linux Bash shell的教程。或者，如果您使用的是较旧版本的Windows，您可以使用Cygwin来获取Bash shell。

打开您的终端并导航到要用作Git存储库的文件夹。出于我们的目的，假设我们在Documents文件夹中有一个名为“MyNovel”的文件夹。请注意，我们的Git回购的单词之间没有空格。如果您这样做会让您的生活更轻松，因为Bash不喜欢空格，而且处理它们会让人感到困惑。

接下来，导航到终端中的MyNovel文件夹。要在Windows 10中执行此操作，命令为：

cd /mnt/c/Users/[YourUserName]/Documents/MyNovel

任何与Windows中保存的文件交互的WSL命令都必须使用/mnt/。另外，请注意，小写的“c”表示您所在的驱动器。如果您的文件位于“D：/”驱动器上，则使用/d/。

对于MacOS和Linux，该命令要简单得多：

cd ~/Documents/MyNovel

从这里开始，命令是相同的。

现在，我们必须将MyNovel文件夹初始化为Git存储库。无论您是刚开始一本新小说，还是已经在里面保存了一些文件，此命令都有效。

git init

您的文件夹现在是Git存储库。不相信我？键入以下内容：

ls -a

该命令要求计算机列出当前文件夹中的所有内容，包括隐藏项目。您应该看到在顶部列出了名为“.git”的内容(请注意句点)。隐藏的“.git”文件夹是保存文档版本历史记录的位置。你应该永远不需要打开这个，但它必须在那里。

第一次提交

在我们进行第一次提交之前，Git想知道您的姓名和电子邮件地址。Git使用此信息来标识谁进行了提交，并且该信息包含在提交日志中。出于实际目的，这并不重要，因为作家通常是单飞的，但Git仍然需要它。

要设置您的电子邮件和地址，请执行以下操作：

git config --global user.email "[Your email]" git config --global user.name "[Your name]"

就这样。现在来看第一个提交。

假设“MyNovel”文件夹中有三个文档：“Chapter1”、“Chapter2”和“Chapter3”。要保存更改，我们必须告诉Git跟踪这些文件。要执行此操作，请键入以下内容：

git add .

句点告诉Git监控文件夹中所有未跟踪的文件(即您想要为其创建历史记录的文件)。该命令还告诉Git准备任何当前跟踪的已更改的文件。此过程称为暂存文件以供提交。

就我们的目的而言，阶段性并不那么重要，但它可以很有用。如果您对第2章和第3章进行了更改，但只想提交第2章中的更改，则应按如下方式暂存第2章：

git add Chapter2.doc

这告诉Git您希望将第2章中的更改准备好提交，但不是第3章。

现在，是第一次提交的时候了：

Git commit -m "This is my first commit."

“-m”称为标志，它告诉Git您想要对消息进行提交和添加，您可以在引号之间看到这一点。我喜欢使用提交消息来标记字数。我还使用它们来记录特殊信息，比如：“这个承诺包括对Acme widgets首席执行官的采访。”

如果我在写一篇故事，我可能会加上这样一条消息：“这个承诺有狗逃跑的新场景。”有帮助的消息使以后更容易找到您的提交。

现在我们已经开始跟踪我们的文档，是时候使用GitHub将我们的作品放到云中了。我使用GitHub作为额外的备份，是查看文档更改的可靠位置，也是访问多台PC上的资料的一种方式。

GitHub入门

首先，你需要在GitHub上注册一个免费账户(创建私人存储库不需要付费账户)。但是，一次私人回购最多只能与三个人协作。如果你有一个五人或更多人的团队在写一篇文章，你需要注册一个专业账户(在撰写本文时，每月7美元)。

创建您的帐户后，让我们进行新的回购。登录到您的帐户并转到https://github.com/new.

我们需要做的第一件事是命名存储库。您可以使用与PC上的文件夹相同的名称。在“Repository name”下，键入“MyNovel”。

“描述”是可选的，但我喜欢使用它。你可以打字，比如，“我关于一个男孩、一个女孩和他们的狗的精彩的新小说”等等。

接下来，选择“Private”单选按钮，但不要选中名为“Initialize this pository with a readme”的复选框。我们不想这样做，因为我们的PC上已经有了存储库。如果我们现在创建一个自述文件，会使事情变得更加困难。

接下来，单击“创建存储库”。在“快速设置-如果您以前做过这种事情”下，复制URL。看起来应该是这样的：

https://github.com/[Your GitHub User Name]/MyNovel.git

现在，让我们回到桌面和我们钟爱的命令行上来。

将您的桌面存储库推送到云

第一次将Repo连接到GitHub时，您必须使用一些专门的命令。第一个问题是：

git remote add origin https://github.com/[Your GitHub User Name]/MyNovel.git

这告诉Git远程存储库是“MyNovel”的起源。然后，URL将Git指向远程源。不要过于纠结于“起源”这个词，它只是一种惯例。如果你愿意，你可以叫它“毛茸茸的”-来源更简单，因为这是使用Git最常见的方式。

当您使用Git上传新的更改时，这称为“推送”。当您下载更改时，它被称为“拉”或“取”。现在，是时候将您的第一个提交推送到GitHub了。下面是您要做的：

git push -u origin master

系统将提示您键入您的GitHub用户名和密码。如果您输入的凭据正确，一切都会上载，您就可以开始工作了。

如果您希望GitHub上传具有更高的安全性，您可以使用SSH密钥。这允许您使用单一密码来上传SSH密钥，因此您不必每次都键入完整的GitHub凭据。另外，只有拥有SSH密钥的人才能上传文件更改。

如果你想了解更多关于SSH密钥的信息，GitHub有关于如何使用它们的完整说明。您还可以将您的Git凭据保存在您的PC上。

就这样!。现在，当您想要提交对文件的更改时，可以使用以下三个简短命令(在导航到“MyNovel”文件夹之后)：

git add .

翻译过来就是：“嘿，Git Stage用于提交所有未跟踪的文件，以及您已经跟踪的文件的新更改。”

git commit -m "1,000 words on the new iPhone review."

翻译过来就是：“嘿，Git，把这些更改和这条消息一起保存。”

git push origin master

翻译为：“嘿，Git，将我在这台电脑上的主副本中的更改上传到GitHub上这个项目的原始版本。”

Git和GitHub奖金提示

差不多就是这样，但这里有一些额外的提示，可以让你的Git和GitHub体验变得更好：

查看过去的提交

要查看过去的提交，请转到GitHub上的MyNovel存储库。在主页顶部的“Code<>”选项卡下，您会看到一个写着“[X]提交”的部分。

单击它，您将看到所有提交的列表。单击所需的提交，您将看到您的文本(如果您以纯文本而不是Word的形式键入文本)。创建提交时，所有以绿色突出显示的内容都是新文本；所有以红色突出显示的内容都已删除。

使用Pull命令

在不同的机器上获取新的存储库很容易。只需导航到要在新机器上保存repo的位置，如CD~/DOCUMENTS。然后，键入：

git pull https://github.com/[Your GitHub User Name]/MyNovel.git

如果出现提示，请键入您的凭据，几秒钟后，您就可以开始工作了。现在，提交新的更改，然后通过git推送原始主机将它们发送回gihub。当您返回到通常工作的PC时，只需打开命令行，导航到您的项目文件夹，然后键入git Pull即可。新的更改将会下载，就像您的编写项目在您的设备上是最新的一样。

不要过河

大多数情况下，写作不是团队的工作，只涉及一个人。正因为如此，本文使用Git的方式不适用于多人项目。具体地说，我们直接对小说的母版进行了编辑，而不是创建所谓的“分支”。分支是小说的实践版本，在这里你可以在不影响原著的情况下进行修改。这就像你的小说有两个不同的副本同时存在，但两个副本都不影响另一个。如果您喜欢Practice分支中的更改，您可以将它们合并到主版本(或主分支)中。如果你不想那样做，也没关系。把练习枝扔掉就行了。

分支非常强大，在一个项目中使用它们将是多个编写器的主要工作流程。在我看来，独立编写器并不真的需要使用分支--只要您不在多台PC上同时对主分支进行不同的更改即可。

例如，您应该在桌面上完成工作，执行提交，然后将更改推送到GitHub。然后，在进行任何进一步编辑之前，打开您的笔记本电脑，将所有新的更改都拉下来。如果不这样做，最终可能会出现Git所说的“冲突”。这时Git会说，“嘿，GitHub和这台PC上的一些更改不匹配。帮我弄清楚这件事。“

解决冲突可能是一件痛苦的事，所以最好尽可能避免冲突。

一旦开始使用Git，您就可以学到很多东西，比如分支、FETCH和PULL之间的区别、GitHub的Pull请求是什么，以及如何处理可怕的冲突。

对于新手来说，Git可能看起来很复杂，但一旦您掌握了它的诀窍，它就是一个强大的工具，您可以使用它来管理和存储您的作品。
